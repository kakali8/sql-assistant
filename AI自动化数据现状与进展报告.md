## 一、项目定位

**一句话：** 让 AI 退一步——用统计方法把数据洞察做到极致（断点检测、显著性检验、跨天连续性追踪），把策略归因和业务决策交还给人。

**核心认知（CEO 评价：所有提案里对 AI 能力边界理解最深刻的）：**
- AI 擅长：异常检测、断点定位、显著性检验、趋势识别、指标联动
- AI 不该做：猜因果归因、给业务建议
- 原因：不是 prompt 不够好，是科学约束（无对照组、多策略并行、样本量小）

### 2.2 关键改进详情

**统计引擎（新增，核心交付）：** `src/ai_analysis/statistical_insights.py`
- 断点检测：PELT 算法（ruptures），输出拐点日期 + 前后均值差 + 方向
- 显著性检验：Mann-Whitney U 检验，输出 p 值 + Cohen's d 效应量
- 连续性追踪：`anomaly_history.json` 记录 first_seen、consecutive_days、状态
- 已有测试覆盖：`tests/test_statistical_insights.py`

**时序窗口扩展（已完成）：**
- `metrics_schema_v2.yaml` 配置 `time_range: last_30_days`
- `pipeline.py` 的 `_prepare_time_window` 按 30 天截取
- 统计引擎要求最少 7 个数据点（`MIN_SERIES_LENGTH = 7`）

**AI 输出重构（已完成）：**
- V3：LLM 做归因 + 给建议 → 输出不可信，被跳过不看
- 当前：LLM 聚焦数据洞察，统计证据嵌入 prompt → 输出可信度提升

**表格展示类型（本周新增）：**
- 新增 `analysis_perspective: table` 指标类型，支持非时序的快照数据展示（如高余额用户列表）
- 数据层：`pipeline.py` 中 table 类型跳过聚合计算，直接导出原始数据为 CSV
- 展示层：`viz_angel_fixed2.py` 新增 `build_table_spec()` 构建表格规格 + JS 端 `data_table` 渲染器
- 支持：可配置列（`table_columns`）、分页（20 行/页）、滚动容器（500px）
- 模块体系从 4 种扩展为 5 种：`series / share / topk / value / table`
- 意义：仪表盘不再局限于图表，可以展示明细数据（用户列表、配置表等），拓宽了数据呈现能力

## 五、AI 直连 ClickHouse 可行性（本次验证）

本次对话中，通过 MCP 工具直连 ClickHouse 执行了「KIX 新用户推送点击率」查询，并与团队人工 SQL 做了对比。

### 结论

| 维度 | AI 表现 |
|------|--------|
| 探索表结构 | ✅ 可行 |
| 简单聚合查询 | ✅ 可行 |
| 涉及埋点语义的业务指标 | ❌ 不可行（status 含义猜错） |
| 复杂业务漏斗 | ❌ 不可行（缺少口径、关联逻辑、过滤规则） |

### 核心差距

AI SQL 将 `status=1` 猜为"点击"，实际点击条件为 `number=2 AND status=2`。还遗漏了：
- `type < 100000` 过滤系统推送
- `hab_app_events.type = push_history.text_id` 关联推送内容
- 17:30 业务日切分
- KIX 用户范围限定

### 建议

当前阶段不建议为 AI 直查 ClickHouse 专门建设。推荐「人写模板 + AI 微调」模式：
- 人定义核心查询的口径和逻辑
- AI 基于已有模板做参数化调整（换日期/维度/筛选）
- 零建设成本，即刻可用

详见：`AI直连ClickHouse可行性评估报告.md`

## 六、当前系统的优势与不足

### 优势

1. **方向正确** — 23/25 的 CEO 评分验证了"AI 退一步"的设计思路
2. **统计引擎已落地** — 断点检测 + 显著性检验 + 连续性追踪，不再靠 LLM 凭感觉
3. **30 天窗口** — 覆盖更多策略时间点，统计方法有足够数据量
4. **双管线架构成熟** — Angel（正向洞察）+ Anomaly（异常检测）互补
5. **自动化端到端** — 数据拉取 → 计算 → AI 分析 → 可视化 → 发布 → 通知，全自动

### 已尝试但效果不佳的方向

1. **统计洞察叙事化** — 已尝试将断点/显著性结果融入报告叙述，发现效果不理想
2. **策略时间线叠加图表** — 已实现但信息过载，对决策帮助有限
3. **连续天数显性展示** — 已实现但实际效果不佳

这三项的共同特点：**技术上可行，但产品体验上没有达到预期**。这正好印证了 v6 的核心认知——难的不是让 AI 做更多，而是判断什么值得做。

### 待探索

4. **数据源仍依赖 Metabase** — 所有数据通过 Metabase API 中转，而非直连 ClickHouse
5. **AI 分析结果未量化验证** — 缺少"统计检测 vs 业务团队实际判断"的准确率对比

## 七、下一步建议

5. **AI 查询模板库** — 积累常用查询的人工模板，让 AI 基于模板做微调

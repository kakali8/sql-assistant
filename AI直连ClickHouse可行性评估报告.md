# AI 直连 ClickHouse 写 SQL 可行性评估报告

> 日期：2026-03-12
> 评估方式：以「KIX 新用户推送点击率」查询为测试案例，对比 AI 生成 SQL 与人工 SQL 的差异

---

## 一、测试背景

通过 MCP 工具将 Claude 直连 ClickHouse 数据库（`new_loops_activity`），要求 AI 完成以下查询任务：

1. 查询 KIX 每日新用户中有多少人点击了 push notification
2. 查询每日推送点击率（push_notification_click_rate）

随后将 AI 生成的 SQL 与团队人工编写的 SQL 进行逐项对比。

---

## 二、AI 查询过程

| 步骤 | AI 行为 | 结果 |
|------|--------|------|
| 1. 列出数据库/表 | 正常 | 成功获取表结构 |
| 2. 获取表字段 schema | 正常 | 成功读取字段名和类型 |
| 3. 理解字段业务含义 | **猜测** | 错误：将 `status=1` 猜为"点击"，实际点击条件为 `number=2 AND status=2` |
| 4. 写出查询 SQL | 结构正确，逻辑错误 | 产出了可执行但**指标不准确**的结果 |
| 5. 获取埋点文档 | Confluence 需登录，无法访问 | 失败 |

---

## 三、核心差异对比

### 3.1 "点击"定义 — 最关键的错误

| | AI SQL | 人工 SQL |
|--|--------|---------|
| 点击判定条件 | `status = 1`（猜测） | `number = 2 AND status = 2`（精确） |
| 数据来源 | 仅 `hab_app_events` | `hab_app_events` JOIN `push_history`（关联推送内容） |
| 过滤规则 | 无 | `type < 100000 AND type IS NOT NULL`（排除系统推送） |
| 用户范围 | 全量用户 | `device_id IN (gametok_user.acc_pid)`（限定 KIX 用户） |

### 3.2 指标深度

| | AI SQL | 人工 SQL |
|--|--------|---------|
| 分析维度 | 仅按天 | 按天 × 推送类型（text_pool） |
| 指标层级 | 单一点击率 | 三层漏斗：发送 → 点击 → 参与PK |
| 业务日期 | 自然日 0:00 切分 | 新加坡时间 17:30 切分（匹配赛事周期） |

### 3.3 数据质量控制

| 维度 | AI SQL | 人工 SQL |
|------|--------|---------|
| 脏数据过滤 | 无 | type < 100000, IS NOT NULL |
| 用户去重 | COUNT(DISTINCT) | GROUP BY ALL 先去重再聚合 |
| 时间窗口 | 硬编码日期 | `now() - interval 30 day` 动态窗口 |
| 表关联 | 未关联推送内容 | type = text_id 关联 push_history |

### 3.4 SQL 结构对比

**AI SQL（2 个 CTE，扁平结构）：**
```sql
WITH sent AS (...), clicked AS (...)
SELECT ... FROM sent LEFT JOIN clicked
```

**人工 SQL（6 个 CTE，分层构建）：**
```
new_user         → 用户池定义
receive_noti     → 事件关联推送内容（核心宽表）
click_agg        → 点击聚合
pn_send          → 发送基数
participated     → PK参与判定（JSON状态解析）
join_after_click → 点击后转化归因
```

## 四、AI 无法从 Schema 推断的业务知识

以下信息仅存在于埋点文档或团队经验中，**无法从表结构逆向推导**：

| 业务知识 | 说明 |
|----------|------|
| `number` 字段含义 | 1=收到推送, 2=用户交互动作 |
| `status` 字段含义 | 1=展示, 2=点击, 3=前台收到 |
| `type` 与 `text_id` 的关联 | `hab_app_events.type = push_history.text_id` |
| `type < 100000` 过滤 | 排除系统级/非业务推送 |
| 业务日切分时间 | 新加坡时间 17:30（匹配赛事运营周期） |
| PK 参与状态判定 | 需解析 `user_status_log` JSON 数组，取最终有效状态 |
| 国家排除逻辑 | SG/HK/CN 需排除（测试/内部用户） |

## 五、可行性评估

### 5.1 AI 直连 ClickHouse 能做什么

| 场景 | 可行性 | 说明 |
|------|--------|------|
| 探索表结构、数据分布 | 可行 | 查看有哪些表/字段、数据量、采样 |
| 字段含义明确的简单查询 | 可行 | 如用户数、注册量、DAU |
| 涉及埋点字段业务含义的查询 | **不可行** | 会猜错，产出不准确的结果 |
| 复杂业务漏斗分析 | **不可行** | 缺少口径定义、关联逻辑、过滤规则 |

### 5.2 改善方案评估

| 方案 | 建设成本 | 维护成本 | 效果 | 建议 |
|------|---------|---------|------|------|
| 把埋点文档喂给 AI | 低 | 中（埋点变更需同步） | 中 | 可尝试 |
| 建物化宽表 | 高 | 高（口径变更需重建） | 高 | 现阶段不建议 |
| 维护口径字典 | 中 | 中 | 中 | 可尝试 |
| **人写模板 + AI 微调** | **低** | **低** | **高** | **推荐** |

### 5.3 推荐方案：模板复用模式

```
人工编写核心查询模板（如推送漏斗 SQL）
         ↓
AI 基于模板做参数化调整（换日期、加筛选、换维度）
         ↓
不需要理解埋点底层逻辑，出错概率低
```

**理由：**
- 当前埋点逻辑复杂且在迭代中（push_notification 口径已从 V1 演进到 V2）
- 核心分析需求偏深度漏斗型，不是简单聚合
- 专门为 AI 建设知识体系的投入产出比不高
- 模板复用模式零建设成本，即刻可用

---

## 六、结论

**AI 直连 ClickHouse 从零写业务 SQL，在当前条件下不现实。** 核心瓶颈不是 SQL 语法能力，而是无法获取埋点字段的业务语义。

**短期建议：** 采用「人写模板 + AI 微调」模式，让人负责口径定义，AI 负责参数调整和批量执行。

**长期如需升级：** 优先将关键事件的埋点字段定义整理成 AI 可读的格式（如 Markdown 文档），逐步扩大 AI 可独立查询的范围。
